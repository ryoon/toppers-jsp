１．マルチプロセッサ・カーネル拡張


1.1 概要

JSPカーネル1.5では,μITRON4.0仕様に準拠を保ちつつマルチ・プロセ
ッサ拡張を行った．拡張を行うに際して,シングル・プロセッサ用
カーネルのターゲット依存部をベースにマルチ・プロセッサ拡張用の
システム関数を追加するだけでマルチ・プロセッサ用カーネルに対応できる方式
とした．
また,マルチ・プロセッサをサポートしないプロセッサ用の従来のシングル・
プロセッサ用カーネルを作成する際は,不要なマルチ・プロセッサ機能は取り
込まないで,最小のカーネルをビルドする形を取った．
マルチ・プロセッサ用のサービス・コールをアプリを,シングル・コア用
カーネルに移し替えて使用しても,問題なく動作するサービス・コール仕様と
した．

1.2 基本的な機能対応用拡張

マルチ・プロセッサ上の排他制御や割込み管理を考慮せす,タスクを純粋に
マルチ・プロセッサで動作させるだけならば,μITRON4.0仕様の属性の
拡張だけで対応を行うことができる．
以下にJSP1.4.5から1.5.0に拡張を行った際の属性の拡張を記載する．

  #define TA_ENAINT     0x01u       /* 割込み要求禁止フラグをクリア */
  #define TA_PRCALL     0x10u       /* 全てのプロセッサで使用 */

  #define TA_PRC1       0x10000u    /* プロセッサ１ */
  #define TA_PRC2       0x20000u    /* プロセッサ２ */
  #define TA_PRC3       0x30000u    /* プロセッサ３ */
  #define TA_PRC4       0x40000u    /* プロセッサ４ */

  #define TPRC_INI      0           /* 初期割付けプロセッサ指定 */

  #define MASTER_PRCID	1       /* マスタプロセッサ(1) */
  #define SUB1_PRCID	2		/* サブプロセッサ(2) */
  #define SUB2_PRCID	3		/* サブプロセッサ(3) */
  #define SUB3_PRCID	4		/* サブプロセッサ(4) */

TA_PRC1～TA_PRC4は,対応のプロセッサを指定する属性で
以下の静的APIやサービス・コールで使用する

CRE_TSK,CRE_CYC,DEF_EXC,DEF_INH,CFG_INT
cre_tsk,cre_cyc,def_exc,def_inh
CFG_INTは,新規に割込みの設定用に拡張した静的APIですが,
必ずしも実装の必要はない
（但し,マルチ・プロセッサではコア毎に割込み制御を行い,
他のコアの割込み設定はできないため,マルチ・プロセッサ用カーネルの割込み
管理には有用な静的APIである）
これ以外の静的APIやサービス・コールはマルチ・プロセッサ用の属性管理は
不要である．
上記の静的APIやサービス・コールで,プロセッサ指定属性を指定しない
場合は,メインプロセッサ(MASTER_PRCID)に割り付けを行う．
TA_PRCALLはプロセッサを指定属性に優先する属性で,
全てのプロセッサ対して設定を行う属性となる．
タスクや周期ハンドラは必ずひとつのプロセッサで実行するため,
CRE_TSK,CRE_CYC,cre_tsk,cre_cycでは,この属性は設定不可となる．

TA_ENAINTはCFG_INT専用属性で,割込み許可状態で起動する指定で
ある．

TPRC_INIはタスクのプロセッサを変更するサービス・コールcfg_prc
（新規）で初期設定のプロセッサを指定するのに使用する．
MASTER_PRCID～SUB3_PRCIDは,前述のcfg_prcやタスクに割り付けら
れたプロセッサを取り出すサービス・コールget_prc（新規）で
プロセッサを認識するIDとして作用する．

1.3 割込みを使った排他制御

マルチ・プロセッサ環境では,割込みはプロセッサ毎に制御するため
シングル・プロセッサ用カーネルでは通常に行われる割込み禁止を
使用したデータの排他制御は,意味を持たない．
マルチ・プロセッサ用環境では,プロセッサや割込みに関係なくデータの
排他制御を行うために,スピンロック機能が必要となる．
μITRON4.0仕様のマルチ・プロセッサ用拡張では,以下の静的APIとサービ
ス・コールを追加した．

CRE_SPN
loc_spn,iloc_spn,unl_spn,iunl_spn

1.4 SYSTICの実装方式

シングル・プロセッサ用カーネルでは,SYSTICにて時間管理を行う,
マルチ・プロセッサ用カーネルでは,マスタプロセッサ（MASTER_PRCID）
用のSYSTICを使って他のプロセッサを管理するSYSTIC_GLOBAL方式
とプロセッサ毎に別々のSYSTICを供給するSYSTIC_LOCAL方式がある
SYSTICの方式により,タスク管理手順等は微妙に異なる,
SYSTIC_LOCAL方式では,プロセッサに割り付けられたSYSTICの割込み
を使ってタスク制御を行うに反して
SYSTIC_GLOBAL方式では,マスタプロセッサ以外のプロセッサで
動作するタスクは,マスタプロセッサのSYSTICより,
プロセッサ間割込みを使って受信したイベントを使ってタスク制御を
行う．
JSP-1.5.0ではSYSTIC_LOCAL方式をデフォルトとする．
但し,TOPPERS_SYSTIM_GLOBALをデファインすればSYSTIC_GLOBAL
方式でビルドを行いカーネルを作成する．

1.5 rot_rdq,irot_rdqの影響範囲

マルチ・プロセッサ用カーネルでは,デフォルト設定ではタスクの優先
順位の回転は発行したプロセッサ上のタスクに対して回転が行われる．
JSP-1.5.0ではSYSTIC_LOCAL方式をデフォルトとする．
但し,TOPPERS_ROTRDQ_GLOBALをデファインすれば,全てのプロセッサ上
のタスクに対して回転要求を発行する方式でビルドを行い
カーネルを作成する．

1.6 ter_tsk

マルチ・プロセッサ用カーネルの場合,μITRON4.0仕様のサービス・
コールで挙動が違うサービスコールが発生する．
特に影響を受けるサービスコールはter_tskである．
ter_tskを発行しても,対象が他のプロセッサ上のタスクで
タスクが実行状態または実行可能状態の場合,E_OBJを返しタスク終了できない．
これを回避するために,終了したいタスクにあらかじめsus_tskを
発行して,実行待ち状態に遷移させておけば,ter_tskにより
終了を行うことができる．
他に影響を受けるサービス・コールについては後述する．

1.7 chg_pid(マルチ・プロセッサ用カーネル専用)

タスクのプロセッサ割り当てを変更する．基本は休止状態のタスクを対象と
するが,TOPPERS_CHGPID_MIGATEコンパイルスイッチを有効にすれば
対象のタスクが他プロセッサ上で実行状態または実行可能状態以外の場合,
マイグレーション処理が可能となる．

 4.8 システム状態管理機能

--------------------------------------------------------------
get_pid    実行状態のプロセッサIDの参照
iget_pid   実行状態のプロセッサIDの参照(非タスクコンテキスト用)
--------------------------------------------------------------
【C言語API】
     ER ercd = get_pid(ID *p_prcid);
     ER ercd = iget_pid(ID *p_prcid);
【パラメータ】
     なし
【リターンパラメータ】
     ER       ercd      正常終了(E_OK)またはエラーコード
     ID       prcid     実行状態のタスクのプロセッサID
【エラーコード】
     E_PAR    パラメータエラー(p_prcidが不正)
     E_CTX    コンテキストエラー
【機能】
 呼び出しを行ったプロセッサのプロセッサID番号を参照し,prcidに
 返す．

--------------------------------------------------------------
chg_pid    休止状態のタスクのプロセッサの変更
--------------------------------------------------------------
【C言語API】
     ER ercd = chg_pid(ID tskid, ID prcid);
【パラメータ】
     ID       tskid     変更対象のタスクID番号
     ID       prcid     タスクに設定するプロセッサID番号
【エラーコード】
     E_ID      不正ID番号(spnidが不正あるいはしようできない)
     E_CTX     コンテキストエラー
     E_OBJ     オブジェクト状態エラー(対象のタスクがDORMANT状態
               でない)
【機能】
 tskidで指定されるタスクの実行プロセッサを,prcidで指定される
 プロセッサに変更する．prcidにTPRC_INI(=0)が指定されると,
 対象タスクのプロセッサを起動時のプロセッサに指定する．
【補足説明】
 デフォルトは指定のタスクは休止状態でなければならない．
 実装依存で, 指定したタスクが実行状態、または、実行可能状態以外の
 場合はマイグレーション処理を行っても構わない．


--------------------------------------------------------------
CRE_SPN    スピンロックの生成（静的API）                  【S】
cre_spn    スピンロックの生成
acre_spn   スピンロックの生成（ID番号自動割付け）
--------------------------------------------------------------
【静的API】
     CRE_SPN(ID spnid, {ATR spnatr} );
【C言語API】
     ER ercd = cre_spn(ID spnid, T_CSPN *pk_cspn);
     ER_ID spnid = acre_spn(T_CSPN *pk_cspn);
【パラメータ】
     ID       spnid     生成対象のスピンロックID番号(acre_spn以外)
     T_CSPN   pk_cspn   スピンロック生成情報を入れたパケットへのポイ
                        ンタ(CRE_SPNではパケットの内容を直接記述する)
    pk_cspnの内容(T_CSPN型)
     ATR      spnatr    スピンロック属性
【リターンパラメータ】
    cre_spnの場合
     ER       ercd      正常終了(E_OK)またはエラーコード
    acre_spnの場合
     ER_ID    spnid     生成したスピンロックのID番号(正の値)または
                        エラーコード
【エラーコード】
     E_ID      不正ID番号(tskidまたはprcidが不正あるいは使用できない)
     E_RSATR   予約属性(spnatrが不正あるいは使用できない)
     E_PAR     パラメータエラー(pk_cspnが不正)
     E_OBJ     オブジェクト状態エラー(対象スピンロックが登録済;
               cre_spnのみ)

【機能】
 spnidで指定されるID番号を持つスピンロックを,pk_cspnで指定される
 スピンロック生成情報に基づいて生成する．spnatrはスピンロックの属性
 である．
 CRE_SPNにおいては,spnidは自動割付け対応整数パラメータ,spnatrは
 プリプロセッサ定数式パラメータである．
 acre_spnは,生成するスピンロックのID番号をスピンロックが登録されて
 いない番号の中から割り付け,割り付けたID番号を返値として返す．
 spnatrに指定できる値とその意味は実装依存である．
 スピンロックはマルチプロセッサにおいて,プロセッサにかかわらず
 タスクと割込み間,または,割込みと割込み間の排他制御を行う機能を
 持つ,従って,同一プロセッサ内では多重にスピンロック状態に
 移行する操作は,E_ILUSEエラーとなる．

--------------------------------------------------------------
loc_spn    スピンロックの状態への移行
iloc_spn
--------------------------------------------------------------
【C言語API】
     ER ercd = loc_spn(ID spnid);
     ER ercd = iloc_spn(ID spnid);
【パラメータ】
     ID       spnid     生成対象のスピンロックID番号
【リターンパラメータ】
     ER       ercd      正常終了(E_OK)またはエラーコード
【エラーコード】
     E_ID      不正ID番号(spnidが不正あるいは使用できない)
     E_CTX     コンテキストエラー
     E_ILUSE   サービスコール不正使用(同一プロセッサで多重にスピン
               ロックを設定した)
【機能】
 spnidで指定されるスピンロックを使用して,スピンロック状態に移行する．
 スピンロック状態では,そのプロセッサの割込を禁止するため,
 そのプロセッサの割込み状態でのスピンロック状態への移行は発生しない．
 スピンロック状態で,他のプロセッサでスピンロック状態に移行しようとした
 場合,最初にスピンロック状態を設定したプロセッサが解除を行うまで
 待ち状態となる．
 同一プロセッサで多重に,スピンロック状態への移行を行ことはできない．
 また,割込みハンドラ等の非タスク状態で,スピンロック状態への移行を
 行った場合,必ず,解除を行なわなければならない．解除を行わない場合,
 実装依存で,警告の表示と停止,または,強制解除を行う機能を追加しても
 構わない．

【シングル・プロセッサ用カーネルの場合】
 CPUロック状態への移行のみ行う

--------------------------------------------------------------
unl_spn    スピンロックの状態の解除
iunl_spn
--------------------------------------------------------------
【C言語API】
     ER ercd = unl_spn(ID spnid);
     ER ercd = iunl_spn(ID spnid);
【パラメータ】
     ID       spnid     生成対象のスピンロックID番号
【リターンパラメータ】
     ER       ercd      正常終了(E_OK)またはエラーコード
【エラーコード】
     E_ID      不正ID番号(spnidが不正あるいは使用できない)
     E_CTX     コンテキストエラー
     E_ILUSE   サービスコール不正使用(同一プロセッサでスピンロック
               への移行が行われていない)
【機能】
 spnidで指定されるスピンロックを使用して,スピンロック状態を解除する．
 スピンロック解除では,そのプロセッサの割込禁止も解除する．
 スピンロック状態でない状態で呼び出された場合,E_ILUSEエラーを返す．

【シングル・プロセッサ用カーネルの場合】
 CPUロック状態の解除のみ行う

 4.9 割込み管理機能

--------------------------------------------------------------
CFG_INT    割込み状態の設定（静的API）                    【S】
cfg_int    割込み状態の設定
--------------------------------------------------------------
【静的API】
     CFG_INT(INTNO intno, {ATR intatr, PRI intpri} );
【C言語API】
     ER ercd = cfg_int(INTNO intno, T_CINT *pk_cint);
【パラメータ】
     ID       intid     定義対象の割込み番号
     T_CSPN   pk_cint   割込み定義情報を入れたパケットへのポインタ
                        (CFG_INTではパケットの内容を直接記述する)
    pk_cintの内容(T_CINT型)
     ATR      intatr    割込み属性
     PRI      intpri    割込み優先度値
【リターンパラメータ】
     ER       ercd      正常終了(E_OK)またはエラーコード
【エラーコード】
     E_RSATR   予約属性(intatrが不正あるいは使用できない)
【機能】
 intnoで指定される割込みに対して,pk_cintで指定される割込み設定に
 基づいて割込み設定を行う．intatrは割込み属性,intpriは割込み優先度
 である．
 intnoの具体的な意味は実装依存であるが,一般的なプロセッサの割込み
 ベクトル番号に対応する．割込みベクトルを持たないプロセッサにおい
 ては,指定することができる割込み番号が一つとなる場合もある．
 pk_cintにNULL(=0)が指定されると,すでに設定済の割込み設定を解除
 する．また,すでに割込み設定済で,再度設定された場合には,以前の
 設定を解除し,新しい設定を行う．
 intatrに指定できる値はTA_ENAINTで,設定時に割込みを許可する．
 他の属性は実装定義とする．
 intpriに設定できる値とその意味は実装定義である．
【スタンダードプロファイル】
 スタンダードプロファイルにCFG_INTをサポートする必要はない．
【マルチ・プロセッサ用カーネルの場合】
 CFG_INT,または,cfg_intを設定していないプロセッサからの
 ena_int,dis_intはE_SYSエラーとなる．
