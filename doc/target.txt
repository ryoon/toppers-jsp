１．ターゲット依存部の開発


1.1 ターゲット依存部の構成

JSPカーネルのターゲット依存部はconfigディレクトリ上に必要なプログラムや
データ等を置く．概要の解説はconfig.txtで行っている．
この資料は作成時の具体的なソース・ファイル情報等を記載する．
最初のディレクトリにプロセッサ依存のリソースを作成する．
このディレクトリはMakefileのCPU変数で指定するディレクトリとなる．
ターゲット・ボード依存部はプロセッサ依存のディレクトリの下に
作成する．MakefileのSYS変数で指定するディレクトリとなる．

1-1-1.ターゲット依存部の機能

JSPカーネルはC言語で記述している．そのためプロセッサのレジスタや
スタックの制御はC言語では記述できない．
この部分を制御するためにアセンブル言語やC言語のインライン・アセンブラ
を使って記述する必要がある．このプログラムはプロセッサ毎に別の
プログラムとなるのでターゲット依存部を設けて専用に記述する必要がある．
以下にターゲット依存部が作成すべき機能について記載する．

(1) スタートアップ・ルーチン

ベアメタル開発と同等に、C言語環境の初期化を行い、ハードウェアの初期化
(hardware_init_hook)を行った後、kernel_start()関数を呼び出し
カーネルを起動する(ベアメタル環境ではmain()関数)．
これらのプログラムを作成を作成する必要がある．
JSPカーネルでは、C++言語環境をサポートしているため、
software_init_hook()で初期化、software_term_hook()で終了後の後処理を
行う、カーネルの起動後のターゲット依存プログラムで行っても問題はない．

(2) カーネルのスケジューリング部のターゲット依存部

JSPカーネルはタスクを制御するために２つの方式を取っている．
・dispatch()関数を用いて実行中のタスクの実行権を放棄する．
・割込み処理中に実行中タスクの実行権を放棄させ、他のタスクに遷移する．
　その情報を以下の３つの変数を使って伝える．
　runtsk：実行中のタスクの情報を持つ構造体(TCB)へのポインタ
　schedtsk：デスパッチ要求があった場合、次に実行すべきタスクのTCBへの
      ポインタ
  reqflg：デスパッチ要求を伝えるBOOL変数、デスパッチ要求があった場合
      TRUEとなる
dispatch関数はスタック操作を行うためアセンブラ言語の記述となる．
runtsk、schedtsk、reqflgの操作は割込み処理で行うため
アセンブラ言語の記述となる．

その他に、TCBのターゲット依存部の定義と実装、タスク起動等の関数

(3) 割込み禁止、許可機能

割込み操作はプロセッサ毎に異なるため、ターゲット依存部で記載する
必要がある．

(4) SYSTICKドライバ

カーネルの時間制御を行うために、システム時間を供給するタイマドライバ
を作成する必要がある．

(5) デバックログ機能用のシリアル・ドライバ

JSPカーネルは種々のデバッグ、実行状態を伝えるためにシステムログ機能を持つ
システムログを表示するために、シリアル通信（UART）を使用する．
そのため、ターゲット・ボードに依存したシリアル・ドライバを作成する
必要がある．

1-1-2.ターゲット機能の実装部

上記のターゲット依存部の機能をCPUディレクトリとSYSディレクトリにて
実装を行う、しかし、プロセッサとターゲット・ボードのハードウェアの
実装方式により、どちらのディレクトリで実装すべきかが変わる．
例えば、ARMのCortex-Mの場合、SYSTICKをCMSISのシステックタイマで
実装するならばCPUディレクトリ、STMのTIM1を使って実装するならば
SYSディレクトリで実装すべきである．

1-2.ターゲット依存部のAPI

JSPカーネルでは、ターゲット依存部の機能を呼び出すために
プロセッサに関係なく、C言語記述のインターフェイス関数を
定めている．


1-2-1. JSPカーネルのターゲット依存部API

タスク制御やタスク状態の設定や参照用のAPIで、関数タイプと
変数タイプがある．

1-2-1-1. スケジュール用のAPI

(1) void dispatch()

実行中のタスクの実行権を放棄して、他のタスクに遷移を要求する関数．

(2) void exit_and_dispatch()

タスクの実行していない状態から、タスク実行を要求する関数．

(3) runtsk/TPCB runtsk

変数、ポインタ、実行中のタスクの情報を持つ構造体(TCB)へのポインタ．
1-5-0以降はTPCBを取得してruntskの設定を行う必要がある．

(4) schedtsk/TPCB schedtsk

変数、ポインタ、デスパッチ要求があった場合、次に実行すべきタスクの
TCBへのポインタ．
1-5-0以降はTPCBを取得してschedtskの設定を行う必要がある．

(5) reqflg/TPCB reqflg

変数、BOOL値、デスパッチ要求があった場合TRUEとなる．
1-5-0以降はTPCBを取得してreqflgの参照と設定を行う必要がある．

(6) enadsp/TPCB enadsp

変数、BOOL値、デスパッチ可能な場合TRUEとなる．
割込みの出口で、reqflgがTRUEであっても、デスパッチは行わず、
reqflgをFALSEにセットする．
1-5-0以降はTPCBを取得してenadspの参照と設定を行う必要がある．
マルチコアの場合、この設定はプロセッサ毎になる．

1-2-1-2. コンテキストAPI

(1) BOOL sence_context()

プロセッサがタスク実行状態ならば、FALSE
割込み処理等でタスク実行状態でなければ、TRUEを返す．

(2) BOOL exc_sense_context(void *p_excinf)

エクセプションが発生した場合のタスクコンテキスト状態を返す．
引数はエクセンプション発生時のスタックポインタ．

(3) BOOL exc_sense_lock(void *p_excinf)

エクセプションが発生した場合のCPUのロック状態を返す．
引数はエクセンプション発生時のスタックポインタ．


1-2-1-3. 割込み許可、禁止API

(1) BOOL sense_lock()

割込み許可の場合、FALSE、禁止の場合、TRUEを返す．

(2) void x_lock_cpu()

コンテキストに関係なく、割込み禁止とする．

(3) void t_lock_cpu()

タスクコンテキストで呼び出す、割込み禁止とする．

(4) void i_lock_cpu()

非タスクコンテキストで呼び出す、割込み禁止とする．

(5) void x_unlock_cpu()

コンテキストに関係なく、割込み許可とする．

(6) void t_unlock_cpu()

タスクコンテキストで呼び出す、割込み許可とする．

(7) void i_unlock_cpu()

非タスクコンテキストで呼び出す、割込み許可とする．

(8) void define_inh(INHNO inhno, FP int_entry)

割込みハンドラの設定

(9) void define_exc(EXCNO excno, FP exc_entry)

例外ハンドラの設定


1-2-1-4. タスク制御構造体(TCB)関係

(1) TCBのタスクコンテキストプロック

TCBのタスクコンテキストに関するデータを定義する．
標準的なプロセッサでは、データはプログラムカウンタとスタック
となる．基本的にCPUディレクトリのcpu_config.hで定義する．
通常は以下の設定となる．

/*
 *  タスクコンテキストブロックの定義
 */
typedef struct task_context_block {
    VP    sp;        /* スタックポインタ */
    FP    pc;        /* プログラムカウンタ */
} CTXB;

(2) TCB中のフィールドの拡張

TCBのタスクの状態を持つtstatのビット数とタスク優先度を持つpriorityの
ビット数を指定する．基本的にCPUディレクトリのcpu_config.hで定義する．
通常は以下の設定となる．

/*
 *  TCB 中のフィールドのビット幅の定義
 */
#define    TBIT_TCB_TSTAT       8    /* tstat フィールドのビット幅 */
#define    TBIT_TCB_PRIORITY    8    /* priority フィールドのビット幅 */

(3) void create_context(TCB *tcb)

タスクが休止状態に移行する時（タスクの生成時，タスクの終了時）に呼
ばれる．基本的には，タスクコンテキストブロックをタスクが起動できる
状態に設定する処理を，create_context と activate_context で行えば
よい．多くの処理はどちらの関数で行ってもよい．
基本的にCPUディレクトリのcpu_context.hで定義する．

(4) void activate_context(TCB *tcb)

タスクの起動準備関数，タスクが休止状態から実行できる状態に移行する
時に呼ばれる．スタック上には何も積まない．
（activate_rではruntskから必要な情報を取り出せるので，
スタックに積む必要はない（効率が悪い））
基本的にCPUディレクトリのcpu_context.hで定義する．

1-2-1-5. マルチコアカーネル用の設定関係

(1) TNUM_PRCID

プロセッサの数，シングルコアカーネルでは1に固定，2以上の設定を行った場合，
カーネルはマルチコアカーネル用のプログラムを生成する．
カーネルライブラリも，シングコアル設定とマルチコア設定では別物となる．
但し、TNUM_PRCIDの数が異なるカーネルライブラリを選定した場合，
ビルド時，または，実行時エラーを発生する．
TNUM_PRCIDが3以上のCPUに対しての検証は行っていない．

(2) TNUM_SIOP

サポートするシリアルI/Oドライバポートの数．

(3) TNUM_PORT

カーネルで使用するシリアルポートの数．
マルチコアカーネルで、プロセッサの数と同数のポート数を設定した場合，
プロセッサ毎に対応したシリアルポートにログ出力を行う．
プロセッサの数を1とした場合，PORT1に全てのログを出力する．

(4) OMIT_SYSTEM_TPCB

このコンパイルスイッチを定義すると，STPCBを生成しない．


1-2-1-6. スピンロック関係(シングルコアカーネルでは不要)

(1) TNUM_SPINLOCKID

最大使用可能なスピンロックの数

(2) KERNEL_GLOCK_SPNLOCK

カーネルのジャイアント・ロックに使用するスピンロックの
ハードウェアアドレス（TNUM_SPINLOCKIDに含まない）

(3) KERNEL_TOOL_SPNLOCK

カーネルのツール・ロックに使用するスピンロックの
ハードウェアアドレス（TNUM_SPINLOCKIDに含まない）

1-2-2. マルチ・プロセッサ・カーネルで追加のAPI

1-2-2-1. プロセッサ・インデックス取得

(1) UINT x_prc_index(void)

プロセッサ・インデックスを取り出す関数

(2) UINT t_prc_index(void)

タスクコンテキストでプロセッサ・インデックスを取り出す関数
基本的にコンテキストによる差異はないため，
#define t_prc_index x_prc_index
で良い

(3) UINT i_prc_index(void)

非タスクコンテキストでプロセッサ・インデックスを取り出す関数
基本的にコンテキストによる差異はないため，
#define i_prc_index x_prc_index
で良い

1-2-2-2. ターゲット依存部追加関数

(1) STPCB *get_sys_stpcb(void)

ターゲット依存部のプロセッサ依存の管理データ(SPCB)を取り出す関数
OMIT_SYSTEM_TPCBコンパイルスイッチを使用すれば,シングルコアの
場合，SPCBを定義する必要はなく，古いソースを使用可能．

(2) void interpro_init(void)

プロセッサ間割込みを初期化する関数，シングルコアカーネルでは不要．

(3) void interpro_term(void)

プロセッサ間割込みを終了する関数，シングルコアカーネルでは不要．

(4) void interpro_int_handler(void)

プロセッサ間割込みハンドラ，シングルコアカーネルでは不要．
コアに関係なく，記載は同一でカーネルに割込みを通知する関数
kernel_ipi_handlerを呼び出す．

1-2-2-3. スピンロックの追加関数

(1) Inline void x_initialize_spin(ID spnid, LOCK *p_spn_lock)

スピンロックの初期化を行うインライン関数．
スピンロック・サービスコールで使用．
spnidで示されるスピンロックのLOCKアドレスをp_spn_lockにセットする．

(2) Inline void x_initialize_giant_lock(LOCK *p_giant_lock)

ジャイアントロックの初期化を行うインライン関数．
ジャイアントロックのLOCKアドレスをp_spn_lockにセットする．

(3) Inline void x_acquire_lock(LOCK *p_lock)

実行状態に関係なく，ジャイアントロックの取得を行うインライン関数．
ジャイアントロックのLOCKアドレスをp_spn_lockにセットする．

(4) Inline void t_acquire_lock(LOCK *p_lock)

タスクコンテキストで，ジャイアントロックの取得を行うインライン関数．
ジャイアントロックのLOCKアドレスをp_spn_lockにセットする．

(5) Inline void i_acquire_lock(LOCK *p_lock)

非タスクコンテキストで，ジャイアントロックの取得を行うインライン関数．
ジャイアントロックのLOCKアドレスをp_spn_lockにセットする．

(6) Inline void x_release_lock(LOCK *p_lock)

実行状態に関係なく，ジャイアントロックの返却を行うインライン関数．
ジャイアントロックのLOCKアドレスをp_spn_lockにセットする．

1-2-3. ツール関係のターゲット依存部API

開発環境依存設定を行うAPI、開発環境はプログラミング言語での設定や
システム・ログ関係の設定がある．
GCCを使用し、サービス・コールを拡張しない場合は変更の必要はない．

1-2-3-1. 開発環境に依存する定義

基本的にCPUディレクトリ上のtool_defs.hで定義する．
コンパイラ依存のデータ型の定義やコンパイラの拡張機能のためのマクロ
定義を行う、マクロ型としては、以下の宣言がある．

・Inline：インライン関数宣言
・Arm：アセンブル宣言
・Weak：関数のWEAK宣言

1-2-3-2. 開発環境依存モジュール

(1) void tool_initialize()

開発環境の初期化を行う関数、特別な開発環境に対応しなければ
未処理で良い．

(2) void call_atexit()

開発環境の終了プログラム、通常はsoftware_term_hook()が定義されて
いれば、これを呼び出す．

(3) トレースログの設定

JSPカーネルは、サービス・コールの実行時と終了時、トレースログ
関数を呼び出す、通常は未処理であるが、トレースログが必要な場合は、
トレースログに出力関数を記載することで、ログ出力が可能となる．

1-3. リネーム機能

カーネルやターゲット依存部の関数や変数が、アプリケーション上の
関数や変数と重なる場合、二重定義となる．
これを防止するために、ターゲット依存部でもアプリケーションで
使用しない名称に変更するリネームがある．
新規にターゲット依存部を作成した場合も、リネーム機能を使って
関数や変数をリネームする必要がある．

CPUディレクトリ上のリネーム設定はcpu_rename.def、
SYSディレクトリ上のリネーム設定はsys_rename.defで定義する．
定義した設定はutilsディレクトリ中のgenrenameのperlスクリプトを
用いて、インクルード・ファイルに変換される．
このインクルード・ファイルはカーネルのビルド設定でインクルード
され、自動的にリネームが行われる．

1-4. アセンブラで参照するタスク情報のオフセット値

ターゲット依存部のアセンブラ言語からTCB中の項目をアクセスした
くても、アセンブラ言語には構造体への参照機能はない．
そのため、構造体の先頭アドレスから項目をアクセスするための
オフセット値を作成する仕組みが必要となる．
JSPカーネルではオフセット値を持つoffset.hを自動生成する機能を
持つ、どのオフセット値を作成するかは、makeoffset.cに定義する．
makeoffset.cもターゲット依存部で用意する必要がある．


1-5. ターゲット依存部ディレトクリ

1-5-1. CPUディレクトリ

CPUディレクトリの一般的なファイルについて解説する．

(1) Makefile.config

MakefileからインクルードされるMakefile
コンパイル変数の設定を行う．

・TARGET：ターゲットのコンパイラを指定する

・KERNEL_ASMOBJS：カーネルで使用するアセンブラ言語の
　　ソースファイル、CPUディレクトリのアセンブラのソースを追加
・KERNEL_COBJS：カーネルで使用するC言語のソースファイル、
    CPUディレクトリのC言語のソースを追加

(2) cpu_defs.h

プロセッサの定義を行う

・プロセッサの定数定義：(例)ARMV7M

・EXCNOの型定義

(3) cpu_config.h

・TCB 関連の定義
・cpu_config.cの関数と変数のプロトタイプ宣言

(4) cpu_config.c

・void cpu_terminate(void)
  プロセッサの終了関数

(5) cpu_context.h

・void create_context(TCB *tcb)
  タスクコンテキストブロックの初期化関数（インライン）
・void activate_context(TCB *tcb)
  タスクの起動準備関数（インライン）

(6) cpu_inst.h

・プロセッサのハートウェア定数
・インラインアセンブル関数

(7) cpu_rename.def

・プロセッサ内のリネーム定義

(8) makeoffet.c

・アセンブラで使用するTCBまたはTINIBの項目へのオフセットを生成する
  ソースファイル、offset.hが生成される

(9) tool_defs.h

・ツールで使用するマクロを定義

(10) tool_config.h

・ツールで使用する関数等を定義

2. カーネルのデバッグ

ソースを作成し、ビルドを行ってjsp.exeを作成しデバッグを行う．
デバッグの手順は、ベアメタル環境での組込み開発と同等であるので
ベアメタル開発を熟知した方には、解説は不要であると思われる．
基本的には、スタートアップ・ルーチンが正しく動くことを確認後、
シリアルドライバsio_init()が、正しく動作できれば、syslog文を
使ってログ出力させながらデバッグを行うのが一般的な開発方法と
なる．
ベアメタル開発で、main()関数を、細かくチェックしながらデバッグ
するように、kernel/startup.c中のkernel_start()を細かく
チェックしながらデバッグして行けばよい．

開発スキルとしては、教育WGで提供している、基礎1実装セミナを
修了しているレベルが必要となる．
